Weak Reference

当一个对象只有Weak Reference引用的时候，允许被GC回收。比如有这样的需求：从网络上下载了一个大型图片，暂时存在内存中。这个图片需要从Activity A传递到Activity B，因为超过1MB，无法通过Intent传递。此时聪明的我想到，反正都是一个进程里的Activity，何必一定要用Intent传递数据。为什么不用一个静态类寄存这个图片然后向外提供访问方法呢？于是我新建了一个BitmapRegistry采用单例模式，Activity A获取单例在其中寄存这个图片，然后Activity B获取单例读取图片。看似很完美，但是却存在隐藏的内存泄露。因为BitmapRegistry持有了静态单例，该单例的生命周期大于Activity的生命周期，当Activity退出时，BitmapRegistry单例仍然持有对图片的强引用，那么GC就不会回收这张图片所占用的内存。应对这种情况，WeakReference就有用武之地了。如果使用WeakReference包装该图片，那么当Activity被销毁时（假设只有这个Activity使用了这个图片），就只剩下了BitmapRegistry对该图片的弱引用，GC仍然会回收这张图片。同理，WeakHashMap就是基于这样的理念，当它的value只有key对他的弱引用时，其对应的value就会允许被GC回收。

Try-With-Resources

某天写代码时看到Android Studio提示可以将try{}finially{}语句自动优化为 try-with-resources语句，我点了一下，发现代码变成了这样：

try (Cursor cursor = getDatabase().rawQuery("..." , null) {
            if (cursor.moveToNext()) {
                return cursor.getString(0);
            }
            return null;
        }

网上搜了以下发现，try括号内的变量要实现AutoClosable接口，就能使用此语句了。

String的深度克隆

今天突然有一个需求，修改一个text后，点击“恢复”按钮，让此text恢复成原来的字符串。我在考虑要不要克隆一个原字符串当作备份用于恢复，于是乎便上网搜索String的克隆，这一搜才知道，String类型和基本类型一样，只要赋值给另一个变量，这个变量就是原变量的深度克隆了，举个例子:

String a=”a”; String b=a; a=”b”;

此时，打印出b，其内容仍然是**”a”。因为String一旦创建后，其内容是不可改变，所有String都会储存进常量池，你对一个String进行赋值，只会让它指向常量池中的一个字符串，而不会让其内容改变。因此，当我们对b赋值为a时，b实际上指向了常量池中“a”的地址，当我们对a赋值为“b”时，a实际上指向了常量池中“b”**的地址，因此，改变a并不会改变b的值，所以b就是实际意义上的a的深度克隆。

Service笔记

关于启动Service的方式有两种，一种是startService，一种是bindService，前者适合那些不用与UI交互的Service，后者适合需要与UI交互的Service。因为bindService可以通过Binder很方便地在Activity与Service之间通讯。startService启动的Service会一直在后台运行，即使启动它的Activity已经销毁；但是bindService所启动的Service会在其Client Activity销毁时一起销毁（假设没有其他client与他绑定）。Service的onCreate方法只会在其启动时执行一次。假如一个Service已经在后台运行，那么再调用bindService或者startService都不会在调用onCreate方法，因此onCreate方法里只适合做一些初始化工作。会调用onStartCommand方法，因此，可以在onStartCommand方法里做一些更新/恢复操作。通过bindService再startService可以实现即能与UI互动，又不会因为Activity被销毁而销毁的Service。
